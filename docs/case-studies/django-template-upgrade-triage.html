Case study

Django template upgrade triage.

A Python web stack migration where framework settings, deployment packages, database adapters, and static files all interact.

Readiness: yellow/red
  • Repo: jpadilla/django-project-template
  • Stars at review: 726
  • Primary risk: Django 3.2 to 6.0 cannot be handled as a single bump
  • Best first action: validate a generated project before changing dependencies
Evidence

What the repository shows.

Django gapPipfile pins Django 3.2.*, while current Django is 6.0.5.
Settings riskThe template uses settings that changed across Django majors, including USE_L10N.
Static riskWhiteNoise middleware and compressed manifest storage need production validation.
Database riskdj-database-url and psycopg2-binary upgrades should be separated from framework semantics.
Deployment riskProcfile, Gunicorn, and WhiteNoise form a deployment batch.
Template riskThe generated project, not only the template repo, must be smoke-tested.
Recommended plan

Upgrade through a stable LTS path first.

1
Generated-project validation PR

Generate a project, run migrations, collect static files, and add a minimal smoke test.

2
Django 4.2 LTS migration

Move from 3.2 to 4.2 first and remove settings that no longer apply.

3
Django ecosystem batch

Upgrade django-configurations, debug toolbar, extensions, and database URL parsing.

4
Deployment package batch

Upgrade Gunicorn, WhiteNoise, PostgreSQL driver, and Werkzeug with production checks.

5
Django 5/6 exploration

Evaluate newer targets only after the LTS migration is green.
